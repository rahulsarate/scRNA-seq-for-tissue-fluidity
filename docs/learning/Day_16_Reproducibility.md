# Day 16: Reproducibility & Environment Management

> **Goal**: Ensure anyone can reproduce every result. Understand seeds, environments, and versioning.

---

## The Reproducibility Crisis

> "I ran the same script and got different results."

This happens because of:
1. **Random seeds**: Algorithms like UMAP and Leiden use random initialization
2. **Package versions**: numpy 1.24 vs 1.26 can give different floating-point results
3. **Missing dependencies**: "Works on my machine" ≠ works everywhere
4. **Implicit state**: Script depends on files generated by another script in a specific order

---

## Our Four Pillars of Reproducibility

### Pillar 1: Fixed Random Seeds

```python
SEED = 42
np.random.seed(SEED)

# Every stochastic function gets the seed:
sc.tl.pca(adata, random_state=SEED)
sc.tl.umap(adata, random_state=SEED)
sc.tl.leiden(adata, random_state=SEED)
```

```r
# R equivalent:
set.seed(42)
```

**Why 42?** Convention (from Hitchhiker's Guide). Any number works — consistency matters.

### Pillar 2: Pinned Environments

```
requirements.txt:      → pip install -r requirements.txt
  scanpy>=1.9
  pandas>=2.0
  
conda YAML:            → conda env create -f scrna_wound_healing.yml
  - scanpy=1.9.6
  - pandas=2.1.4
```

**Note**: `>=1.9` allows updates (flexible). `=1.9.6` locks exact version (strict). Use strict for final published analysis.

### Pillar 3: Config-Driven Parameters

```yaml
# configs/analysis_config.yaml
# Every parameter documented in one file:
qc_thresholds:
  min_genes_per_cell: 200
  max_percent_mt: 15
clustering:
  default_resolution: 0.8
  n_pcs: 30
```

No hunting through code for hardcoded values.

### Pillar 4: Version Logging

```python
# At the end of every script:
import scanpy as sc
print(f"Scanpy version: {sc.__version__}")
print(f"AnnData version: {ad.__version__}")
print(f"numpy version: {np.__version__}")
```

```r
sessionInfo()  # Prints ALL loaded package versions
```

## Environment Deep Dive

### requirements.txt vs conda YAML

| Feature | requirements.txt | conda YAML |
|---------|-----------------|------------|
| Manager | pip | conda |
| Python packages | ✓ | ✓ |
| C/Fortran libraries | ✗ (sometimes fails) | ✓ |
| System deps (HDF5) | ✗ | ✓ |
| Speed to install | Fast | Slower |
| Isolation | venv directory | Named environment |
| Best for | Development | Production/sharing |

### Setting Up From Scratch

```bash
# Option 1: pip
python -m venv .venv
.venv\Scripts\activate        # Windows
pip install -r requirements.txt

# Option 2: conda
conda env create -f configs/conda_envs/scrna_wound_healing.yml
conda activate scrna_wound_healing
```

## Interview Q&A

### Q: "How do you ensure reproducibility?"

> "Four mechanisms: (1) Fixed seed 42 in every stochastic step. (2) Pinned package versions in requirements.txt and conda YAML. (3) All parameters in a central YAML config — nothing hardcoded. (4) Immutable raw data. Anyone cloning the repo can recreate every result by: install environment → run scripts in numbered order → compare outputs."

### Q: "What if a collaborator gets different results?"

> "First I check: same random seed? Same package versions? Same input data? Same config? The most common cause is package version drift — a numpy update changing floating-point behavior. That's why strict version pinning matters for final publications."

## Self-Check Questions

1. **Why set random seeds?** → Algorithms like UMAP/Leiden use random initialization; seeds make them deterministic
2. **What seed do we use?** → 42
3. **What does requirements.txt do?** → Lists Python packages for pip install
4. **Why both pip and conda?** → pip for speed/development; conda for compiled dependencies
5. **Where do all parameters live?** → configs/analysis_config.yaml
6. **How do you log package versions in R?** → sessionInfo()
7. **Why number scripts (01_, 02_)?** → Clear execution order
8. **What makes raw data immutable?** → data/raw/ is read-only by policy; scripts never write there
9. **What's version pinning?** → Specifying exact package versions (e.g., scanpy=1.9.6)
10. **Clone → install → run → same result?** → That's the goal of reproducibility
